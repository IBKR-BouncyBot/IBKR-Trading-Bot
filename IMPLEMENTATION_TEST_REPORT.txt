IBKR Trading Bot 3.0.18 - Event-Driven Cadences Implementation and Test Report
Date: 2026-07-18

SCOPE
-----
Implemented the requested performance changes:

1. Replaced the fixed one-second controller-loop delay.
2. Removed the unconditional 250 ms market-price wait.
3. Separated broker, strategy, GUI, database-snapshot, and maintenance work onto independent monotonic cadences.
4. Released the behavior-changing work as v3.0.18 rather than leaving it under v3.0.17 metadata.

IMPLEMENTATION
--------------
Controller scheduling
- The worker no longer calls time.sleep(1.0).
- Queue.get(timeout=...) provides an interruptible wait: GUI commands wake the worker immediately.
- Independent default cadences are:
  - broker callbacks/connectivity: 0.05 seconds
  - strategy evaluation/order polling: 0.10 seconds
  - GUI snapshots: 0.50 seconds
  - read-heavy database snapshot data: 1.00 second
  - maintenance scheduling: 1.00 second
- The next wait is calculated from the earliest monotonic deadline and capped by MAX_IDLE_WAIT_SECONDS.
- Broker callbacks and connectivity are processed before strategy evaluation.
- A failed/unready broker cycle prevents the associated strategy cycle.
- Command bursts are bounded so periodic responsibilities regain control.
- Shutdown checks the stop event immediately after queue wake and does not execute an older queued broker command after shutdown begins.

Market data
- The initial subscription snapshot is inspected before any wait.
- Scheduled strategy reads use timeout=0 and return cached subscription state without sleeping.
- Explicit operations that intentionally allow a positive timeout wait only while data is absent, in slices no larger than 50 ms.
- Changing the TWS market-data type no longer inserts an unconditional 250 ms sleep.
- Nonblocking auto-mode reads use the currently selected/cached concrete mode and do not synchronously qualify new contract variants.

Order polling
- Periodic cached order polling no longer sleeps for 100 ms.
- A cache miss may request an open-order refresh, but the periodic path waits for later broker callbacks rather than blocking the strategy cadence.
- Explicit recovery/cancel/open-order operations retain bounded waits where synchronous confirmation is part of the operator action.

Database and maintenance separation
- GUI snapshots consume a database snapshot cache refreshed on the database cadence.
- Recent events, history summaries, and GUI-only risk/position facts are no longer reread on every GUI render.
- Critical persistence remains synchronous: order intent, order identifiers, fills, cancellations, state transitions, reconciliation, recovery, and shutdown checkpoints were not moved behind a delayed cache.
- Final BUY authorization and risk preflight continue to query live SQLite state; cached GUI facts cannot authorize an order.
- Human-readable debug-report writing and stale-cycle housekeeping are owned by the maintenance cadence rather than GUI rendering.

THREADING AND SAFETY BOUNDARY
-----------------------------
- Broker callbacks, commands, strategy evaluation, and critical persistence remain serialized on the existing single worker thread.
- No parallel access to the broker adapter was introduced.
- Connectivity callbacks are pumped before command dispatch to preserve fail-closed submission behavior.
- The GUI/database cache affects display work only, not live order authorization.
