Implementation and verification report

BASELINE
========

This release was produced from the corrected v3.1.2 source archive:

  IBKR_Trading_Bot_3.1.2_Fill_Reconciliation_Isolation_Stage3_Close_Ruff_Fixed_Source.zip

The v3.1.2 fill reconciliation, late execution/commission handling, strict
full-OrderRef ownership, diagnostic throttling, execution-time correction,
Stage-3 profitable pre-close liquidation, documentation, tests, and portable
build layout were retained. The release version was advanced to 3.2.0.

IMPLEMENTED SCOPE
=================

1. EXACT USD/EUR ORDINARY-STOCK CONTRACTS OVER SMART

- Supported contract currencies are USD and EUR.
- Supported security type is ordinary STK only.
- Routing is SMART only.
- Production use requires an exact positive IBKR conId selected from the API
  contract-search results. Editing the ticker invalidates the prior selection.
- Qualification rechecks conId, symbol, currency, security type, SMART route,
  primary exchange, and contract details before monitoring or order submission.
- ContractDetails capability checks require SMART availability plus the MKT and
  TRAIL order types used by BouncyBot when those broker fields are supplied.
- Existing price-dependent market-rule normalization remains authoritative.
- Whole-share minimum-size and size-increment metadata is validated. BUY size
  may be rounded down before intent persistence; SELL size is never rounded
  down because doing so could leave an untracked app-owned remainder.

2. ONE CONTRACT CURRENCY PER PORTABLE SQLITE DATABASE

- The existing app_settings table stores a database_contract_currency key.
- A zero-cycle draft database may switch between USD and EUR.
- The first persisted cycle locks the database currency.
- Existing v3.1.2 databases infer the lock from historical cycles. Legacy blank
  cycle currencies are interpreted as USD because earlier supported releases
  were USD-only.
- Mixed-currency evidence fails closed. BouncyBot does not perform FX
  conversion for budgets, P/L, risk totals, or reinvestment.
- A non-zero commission reported in a different currency is retained as a raw
  broker fact but excluded from local net P/L. Auto-repeat is disabled and a
  decision event identifies the mismatch.
- Position, risk, and reinvestment queries use the exact selected conId while
  retaining conservative compatibility with legacy same-ticker rows that lack
  a conId. Live account-position lookup does not fall back to another positive
  conId merely because the ticker symbol matches.

3. CONTRACT-SPECIFIC SESSION AND QUANTITY SAFETY

- Date-specific liquidHours and timeZoneId from the exact contract drive RTH
  status and timing guards.
- The historical New York fallback is allowed only for recognized U.S. equity
  primary exchanges.
- A non-U.S. contract with missing, invalid, or unparseable session metadata
  fails closed instead of inheriting U.S. hours.
- GUI currency formatting, previews, confirmations, Trade History, audit views,
  and example data are contract-currency aware.

4. INDEFINITE FIXED-CADENCE LOCAL API RECONNECT

- A user-requested Connect enables automatic retries before the first socket
  attempt, so an initial failure does not require another click.
- A lost local TWS/IB Gateway API socket is retried every 10 seconds.
- There is no retry limit and no exponential backoff.
- A successful reconnect resets the failure count and invokes the existing
  order, execution, position, market-data, and recovery reconciliation paths.
- Manual Disconnect and application shutdown disable retries.
- A healthy local API socket with a separate Gateway/TWS-to-IBKR server outage
  remains under the existing upstream-connectivity recovery logic and is not
  repeatedly torn down.

5. SAME-RELEASE BUGSCAN CORRECTIONS

- Qualification now verifies the broker-returned symbol, SMART order route, and
  selected primary exchange instead of relying on conId/currency/security type
  alone. The selected primary exchange is retained if IBKR returns that field
  blank.
- SMART/direct market-data fallback variants use the qualified contract's EUR
  or USD currency when the raw contract object omits currency. EUR contracts no
  longer produce a USD fallback request or cache key.
- Resume checkpoints validate/claim database currency in the same SQLite
  transaction as the settings and active cycle, closing a shutdown-only path
  that could otherwise bypass the one-currency lock.
- Reconnect attempts now observe the full ten-second interval even when a
  deterministic monotonic clock starts at 0.0.
- An exactly zero commission is currency-neutral. Non-zero cross-currency
  mismatch events are deduplicated persistently across controller restarts,
  including events written by the earlier v3.2.0 build without a dedupe_key.
- The two supplied Ruff 0.16.0 I001 findings were corrected by using the exact
  required import-boundary spacing in app/ib_adapter.py and app/storage.py. No
  imports or runtime bindings changed.

DATABASE COMPATIBILITY
======================

v3.2.0 adds no SQLite table or column. The currency lock uses the existing
app_settings table and is created lazily.

A database created by the corrected v3.1.2 source was populated with:

- saved connection and strategy settings;
- one active Stage-4 cycle with an exact conId;
- one completed BUY and one working SELL order record;
- one BUY execution; and
- one audit event.

Opening that database with v3.2.0 preserved all tables, columns, indexes, core
rows, settings, cycle state, orders, execution, and audit event byte-for-byte at
the SQLite value level. One app_settings row was added with the inferred USD
currency lock. The compatibility check passed.

This validation is not a claim of exhaustive branch, timing-interleaving,
