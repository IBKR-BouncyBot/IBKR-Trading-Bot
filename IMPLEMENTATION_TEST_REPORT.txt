BASELINE

PROBLEM EVIDENCE
================

The supplied IB Gateway log showed two independent broker-boundary failures:

1. An IREN what-if BUY TRAIL request was rejected because the request used
   whatIf=True with transmit=False.
2. The following live IREN BUY TRAIL used a four-decimal stop price derived
   from ContractDetails.minTick=0.0001 and was rejected with IBKR error 201,
   "Order rejected - reason: Invalid Price".

The same log showed a successful NBIS BUY TRAIL, demonstrating that the account,
Gateway connection, and general native trailing-order mechanism were working.

IMPLEMENTED CHANGES
===================

1. MARKET-RULE PRICE NORMALIZATION

The live IBKR adapter now:

- reads validExchanges and the positionally corresponding marketRuleIds;
- preserves empty rule positions so an exchange cannot inherit another route's
  rule accidentally;
- selects the rule for the requested route, normally SMART;
- requests price bands through reqMarketRule;
- selects the increment applicable at the proposed price;
- rounds BUY stop/sizing prices upward and SELL stop prices downward with exact
  decimal arithmetic;
- re-evaluates the band if rounding crosses a price boundary;
- caches successful contract-detail and market-rule results for the adapter
  session; and
- fails closed before order intent/submission when IBKR advertises a market rule
  but that rule cannot be selected or loaded.

The established contract-level minimum-tick fallback remains available only for
adapters/contracts that do not advertise a market rule.

2. STRICT WHAT-IF VALIDATION

What-if checks now use IB.whatIfOrder with whatIf=True and transmit=True.
The result fails closed for missing OrderState, validation/rejection/error
statuses, rejection warning text, unset-value sentinels, or a response with no
usable margin/equity result. Legitimate numeric zero changes remain valid.

A ValidationError can no longer be reported as a passing margin check.

3. BROKER-ORDER ERROR RETENTION

Order-related IBKR callbacks for app-owned orders now retain:

- error code and message;
- request/order ID;
- OrderRef and permanent order ID when available;
- ticker and currency;
- advanced rejection JSON when supplied; and
- callback timestamp and diagnostic representation.

The callback race in which IBKR reports an error before local trade registration
is handled through a 30-second, 256-entry bounded cache. Only definitive
order-validation codes or clearly order-specific messages enter that unbound
cache. Manual orders and unrelated request errors are not attributed to the bot.

The controller persists the error in broker_events, adds a BROKER_ORDER_ERROR
decision event, and exposes the broker reason through the existing status/audit
interfaces.

4. NO-FILL REJECTION CIRCUIT BREAKER

An unfilled BUY order that becomes Inactive or Rejected no longer resets to
Stage 1 for repeated automatic attempts. A terminal no-fill state carrying a
substantive broker validation/rejection error also stops the cycle.

The cycle enters ERROR/manual review and records that no replacement or fresh
cycle retry will be submitted. A normal Cancelled or ApiCancelled state without
substantive rejection remains a normal Stage-2 cancellation and resets to
Stage 1. IBKR code 202 alone remains an ordinary cancellation notification.

SCOPE AND COMPATIBILITY
=======================

- No SQLite schema change was introduced in v3.1.1.
- A database created by v3.1.0 with saved connection/strategy settings, an
  active Stage-4 cycle, a working SELL order, an execution, and an audit event
  was opened by v3.1.1 successfully.
- Tables, columns, indexes, row counts, saved settings, and cycle fields were
  unchanged by v3.1.1 initialization.
- Existing v3.1.0 and v3.0.19 databases remain forward-compatible.
- ATR calculations, strategy percentages, order ownership, quantity/P&L
  calculations, market-data capture, Stage-4 close-before-RTH liquidation, and
  recovery policy were not otherwise changed.

EXTERNAL VALIDATION STILL REQUIRED
==================================

Ruff and Pyright were not installed in the offline Linux validation environment,
so no local v3.1.1 Ruff or Pyright pass is claimed. The Windows run_all_tests.bat
gate must be run before producing the production executable.

The following also remain external checks:

- native Windows PyInstaller build and launch;
- actual PySide6 rendering and interaction on Windows;
- live TWS/IB Gateway market-rule and error-callback behavior;
- paper-account what-if and order lifecycle testing, including IREN; and
- production market-data, entitlement, pacing, fill, commission, and latency
  behavior.

This release should first be exercised with IBKR paper trading. A passing offline
suite cannot guarantee broker acceptance, execution price, or exchange behavior.
