Requested production change
---------------------------
One optional Stage-4 exit policy was added under Risk and timing:

    Cancel SELL trail and liquidate before close
    Liquidate before close: 5 min

The checkbox defaults to OFF. The minute value defaults to 5 and accepts 1
through 240. The controls and information badges use the same tooltip and input
locking conventions as the surrounding interface.

Exact behavior
--------------
The policy applies only to the normal app-owned final SELL_TRAIL while the cycle
is in Stage 4 (SELL_TRAIL_ACTIVE). It does not change Stage 1, Stage 2, Stage 3,
protective SELLs, ATR calculations, trailing percentages, sizing, or normal
order construction.

At the configured contract-specific RTH cutoff, while RTH is confirmed open:

1. BouncyBot persists that the automatic close workflow has started.
2. It requests cancellation of the final SELL trail once.
3. It continues polling the original order and recording fills.
4. It submits no replacement until IBKR reports a terminal original-order state.
5. It totals persisted final-SELL executions across the original trail and any
   replacement.
6. If shares remain, it submits one SELL MKT order for only that app-owned
   remainder, with TIF DAY and outsideRth=False.
7. It reaches Stage 5 only after cumulative SELL fills equal the app-owned BUY
   quantity. Normal commission, P/L, capture, history, and Auto-repeat handling
   then continue.

The trailing percentage is never changed to zero.

Safety and failure behavior
---------------------------
- A complete fill during the cancellation race completes normally and suppresses
  the replacement.
- A partial original fill reduces the replacement quantity.
- An original order that may still be working remains the only SELL; no second
  SELL is submitted.
- If cancellation is not confirmed before RTH closes, the original order remains
  the only app exit order and the bot continues monitoring it.
- If cancellation becomes terminal after close, no replacement is submitted.
- Submission failure, replacement rejection/cancellation/inactivity, an
  incomplete replacement at the RTH boundary, an invalid app-owned quantity, or
  a possible over-sell moves the cycle to ERROR/manual review.
- No outside-RTH fallback is submitted and the cancelled trail is not silently
  recreated.
- The manual Stop-screen market-close action is blocked while this automatic
  workflow is active, preventing a second app-created market SELL.
- Workflow settings and runtime flags are persisted in additive SQLite columns
  and are included in reconnect/restart reconciliation.

Coverage collection note
------------------------
Coverage was collected in isolated chunks because the complete suite under one
instrumented process was impractically slow on the Linux validation host. The
two timing-sensitive tests in tests/test_v300_large_database_performance.py were
run separately without coverage instrumentation and both passed. All 861 tests
were still executed in the complete plain pytest run.

External checks not claimed
---------------------------
Ruff and Pyright were unavailable in the offline Linux environment and could not
be installed without network access. The Windows run_all_tests.bat quality gate
therefore remains required before producing the release executable.

A native Windows PyInstaller launch, clean Windows-machine smoke test, and live
TWS/IB Gateway paper-account cancellation/fill exercise were also outside this
environment. Those checks are necessary before production use because broker
cancellation timing and exchange fills cannot be proven by deterministic tests.
