BouncyBot - IBKR Portable Trading Bot v3.1.2
Fill reconciliation, strict order ownership, diagnostics, timestamps, and Stage-3 close
Implementation and verification report
Date: 2026-07-23

BASELINE
========

This release was produced from the corrected v3.1.1 source archive:

  IBKR_Trading_Bot_3.1.1_IBKR_Order_Validation_Ruff_Corrected_Source.zip

The v3.1.1 market-rule normalization, what-if validation, broker-error
retention, rejection circuit breaker, documentation, tests, and portable build
layout were retained. The release version was advanced to 3.1.2.

PROBLEM EVIDENCE
================

The implementation addresses defects established by comparing two live audit
bundles and the associated Gateway evidence:

1. A BUY could partially fill, trigger cancellation of the remainder, receive a
   second fill during the cancellation race, and leave that second execution
   outside the cycle's durable app-owned quantity.
2. Late execDetails and commission callbacks were audited but were not always
   applied to the cycle ledger or completed-cycle P/L.
3. A shared Master-client feed could associate another portable instance's
   IBKRBOT-prefixed callback with the current active cycle despite a different
   complete OrderRef.
4. Native trailing diagnostics changed their throttle key whenever a displayed
   price changed, so routine waiting diagnostics could be persisted every tick.
5. Broker execution timestamps could be shifted by the host timezone.
6. The existing close-before-RTH option covered Stage 4 but not a profitable
   Stage-3 position that had not yet reached the normal SELL activation level.

IMPLEMENTED CHANGES
===================

1. BUY SETTLEMENT REMAINS IN STAGE 2 UNTIL TERMINAL

- A positive partial BUY fill no longer moves the strategy immediately to
  Stage 3.
- The original BUY order remains under broker polling until IBKR reports a
  terminal status.
- The unfilled remainder is cancelled once; the one-shot cancellation state is
  persisted as buy_remainder_cancel_requested.
- If the cancellation call fails before confirmation, the one-shot flag is
  cleared so a later broker cadence can retry.
- Additional fills during cancellation are reconciled into the cumulative
  quantity, average price, and commission.
- Stage 3 and any protective SELL are entered/created only after terminal BUY
  settlement.
- Restart and reconnect recovery use the same settlement path.

2. IDEMPOTENT LATE EXECUTION AND COMMISSION CALLBACKS

- Real executions are keyed by exact IBKR execution ID and upserted rather than
  appended blindly.
- Replayed execDetails callbacks do not duplicate quantity.
- Commission can arrive before or after execution details.
- An explicit commissionReport callback overrides an ib_async default 0.0
  commission placeholder.
- A stable residual cumulative placeholder represents order-status quantity or
  commission that has not yet arrived under individual execution IDs.
- The placeholder shrinks as real callbacks arrive and is removed when the
  individual ledger fully represents the broker cumulative state.
- Cycle quantities, weighted average prices, commissions, and P/L are projected
  from the durable ledger monotonically; an incomplete callback subset cannot
  reduce a larger broker cumulative quantity.
- Late commissions update completed-cycle net P/L.
- Quantity conflicts fail closed: late BUY quantity after creation of an exit
  order and SELL quantity above app-owned BUY quantity move the cycle to ERROR
  for manual review.

3. STRICT ORDER-REFERENCE OWNERSHIP

- The generic IBKRBOT prefix is no longer accepted as proof that an order belongs
  to this portable instance.
- Broker callbacks are attached to a cycle only when their complete OrderRef
  exactly matches an OrderRef persisted in the local database.
- Unmatched Master-feed events remain unowned broker diagnostics with
  cycle_id=NULL.
- Unmatched events cannot alter the active cycle, quantities, P/L, current
  status, decision stream, or recovery cancellation set.
- No installation-specific identifier was added, as requested.
- Trade-off: loss or replacement of the local database can require manual
  reconciliation because broad prefix-based recovery is intentionally rejected.

4. STABLE NATIVE-TRAIL DIAGNOSTIC THROTTLE

- The throttle identity is now based on cycle ID, side, and exact order
  reference rather than the complete message text.
- Numeric diagnostic values can change without creating a new throttle bucket.
- Repeated routine waiting diagnostics are limited to the configured interval.
- The existing bounded warning-cache behavior remains in place.

5. EXECUTION TIMESTAMP CORRECTION

- Live execDetails uses ib_async's aware fill-receipt timestamp as the canonical
  persisted UTC execution timeline.
- The broker-decoded execution timestamp is preserved separately in raw/audit
  data.
- Recovered historical fills use the decoded broker execution timestamp.
- The adapter no longer forces TimezoneTWS to UTC.
- Tests cover aware live timestamps, recovery timestamps, and host-timezone
  independence.

6. PROFITABLE STAGE-3 CLOSE BEFORE RTH

The existing setting remains:

  Cancel SELL trail and liquidate before close

It remains OFF by default and retains the existing configurable cutoff.

When enabled in Stage 3, BouncyBot starts the workflow only when:

- RTH is confirmed open;
- the contract-specific cutoff has been reached;
- a fresh selected strategy price is available; and
- selected current price is strictly greater than weighted average BUY price.

Commissions are deliberately ignored for this eligibility comparison.

If no protective SELL is working, the bot submits one RTH-only DAY market SELL
for the app-owned unsold quantity. If a protective SELL is working, the bot:

1. requests cancellation once;
2. waits for terminal broker confirmation;
3. applies any fills during cancellation;
4. recalculates remaining app-owned quantity;
5. rechecks that the selected price is still strictly above average BUY; and
6. submits only the remaining quantity.

If the price no longer passes after cancellation, the cycle moves to ERROR for
manual review rather than transmitting the replacement. A qualifying quote does
not guarantee a profitable fill: the market order can execute below the checked
price or average BUY price. No outside-RTH fallback is submitted. Existing
Stage-4 cancel-confirm-replace behavior remains intact.

DATABASE COMPATIBILITY
======================

v3.1.2 adds one additive, defaulted cycles column:

  buy_remainder_cancel_requested INTEGER NOT NULL DEFAULT 0

A representative v3.1.1 SQLite database containing settings, an active Stage-4
cycle, an order, an execution, and an audit event was opened by v3.1.2. Existing
settings and cycle data remained readable; the new field defaulted to false.
No table was removed, renamed, or rewritten. The migration remains idempotent.

QUALITY AND EXTERNAL LIMITS
===========================

Ruff and Pyright were not installed in the offline Linux release environment, so
this report does not claim a new local Ruff or Pyright pass. Import blocks were
kept in conventional Ruff/isort order and Python compilation/AST checks passed.
The Windows run_all_tests.bat gate remains required before producing the
production executable.

The automated suite uses deterministic adapters and does not prove live IBKR
behavior. A Windows PyInstaller build, clean-machine launch, and TWS/IB Gateway
paper-account exercise remain external checks. The paper exercise should include
at least one partial BUY/cancel race, callback replay/commission arrival, shared
Master-feed observation, and Stage-3 close-before-RTH path.

This validation is not a claim of exhaustive branch, timing-interleaving,
exchange, or broker coverage. It verifies the complete repository test
collection and the specific failure modes implemented in v3.1.2.
